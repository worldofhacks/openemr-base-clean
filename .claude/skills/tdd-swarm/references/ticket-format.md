# Ticket Format

One markdown file per ticket in `tickets/`. The ticket file IS the sub-agent's
task brief — write it so a fresh agent with zero session context can execute it.
Local files are the source of truth; GitHub Issues are a mirror.

## File: `tickets/T-<nnn>.md`

```markdown
---
id: T-014
title: Medication list tool returns sourced, structured results
status: backlog        # backlog | tests-written | in-progress | blocked | review-passed | done
wave: 2
depends_on: [T-006, T-009]
branch: ticket/T-014-medication-tool
file_scopes:
  - src/agent/tools/medications.*
  - src/agent/schemas/medication.*
test_scopes:
  - tests/unit/tools/medications.*
  - tests/integration/agent/medication-flow.*
github_issue: 47
model_hint: cheap      # cheap | standard | capable — planner's complexity estimate
attempts: 0
traces_to:             # every ticket serves something written — or it doesn't exist
  - PRD §4.2 (medication reconciliation)
  - USERS.md UC-3 (pre-visit medication review)
  - ARCHITECTURE.md §2.3 (tool layer)
---

## Context
One paragraph: where this fits in the architecture, which interfaces from
completed tickets it consumes (with signatures), and any decisions already made.

## Acceptance Criteria
- **AC-1**: Given a patient with 3 active medications, when the tool is called with
  the patient id, then it returns all 3 with dosage, route, and a source record id each.
- **AC-2**: Given a patient with no medications, when called, then it returns an empty
  list — not an error, not null.
- **AC-3**: Given an unauthorized requester, when called, then it returns AUTHZ_DENIED
  and logs the attempt with the correlation id.

Ids are stable — tests cite them (`spec(T-014:AC-2)`) and spec-lint enforces
that every AC has a test and every test has an AC.

## Test Plan
- Unit (deterministic): schema validation, empty case, authz rejection
- Integration (deterministic): tool call through the agent loop with demo EHR data
- Eval (non-deterministic, LLM behavior): response cites source record ids for
  every medication claim — eval harness case, graded, threshold ≥ 0.95
- E2E: none (covered by T-021)

Deterministic rows become frozen tests. `Eval` rows become eval-suite cases —
graded thresholds, never brittle exact-match against sampled LLM output.

## Definition of Done
- [ ] All acceptance criteria have a passing test written by the Test Agent
- [ ] Local gates green (see quality-gates.md)
- [ ] Tool input/output has a strict schema; schema is the contract
- [ ] Every response field traceable to a source record (no inferred values)
- [ ] No secrets, no PHI in logs
- [ ] Reviewer + Security agents approved

## Out of Scope
What this ticket explicitly does NOT do (stops implementer scope creep).
```

## Rules

- **Small**: ≤ ~half a day of agent work. If the DoD has more than ~8 items or
  criteria span two concerns, split it.
- **File scopes are exclusive within a wave.** Overlapping scopes = forced
  dependency. This is what makes parallel dispatch safe.
- **Status transitions** (only the orchestrator writes them):
  `backlog → tests-written → in-progress → review-passed → done`,
  with `blocked` reachable from any active state. Every transition appends a
  ledger line and updates the GitHub issue.
- `attempts` is incremented by the orchestrator per implementation loop; at 3,
  status becomes `blocked` automatically.

## `TICKETS.md` index (repo root)

Regenerated by the orchestrator on every status change:


## GitHub Issues sync

Mirror via `gh` CLI (or the GitHub MCP if connected):

- Ticket created → `gh issue create --title "T-014: <title>" --body-file tickets/T-014.md --label tdd-swarm,wave-2`
- Status change → `gh issue comment <n> --body "status: in-progress (attempt 2)"`
- Blocked → add label `blocked` (this is the human's signal to look)
- Done → `gh issue close <n> --comment "merged in wave 2: <commit range>"`

Sync is one-directional (local → GitHub) during a run. If a human edits an
issue mid-run, the orchestrator flags the divergence rather than silently
overwriting either side.
