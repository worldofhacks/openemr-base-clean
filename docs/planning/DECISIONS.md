## D1. Target user: PCP with 20-patient day — **locked**
**Options:** PCP / ED resident / hospitalist.
**Why (verified, R10):** three independent grounds. (1) **Platform:** OpenEMR's core is ambulatory practice management (scheduling, office encounters, billing — see its controller surface); it has no inpatient census/ADT workflow, so a hospitalist persona fights the platform. (2) **Demo data:** the repo's own sanctioned sample-data tool imports **Synthea** patients, and Synthea is *by design* modeled on the most frequent primary-care encounter reasons + highest-morbidity chronic conditions — longitudinal records with exactly the classes the pre-visit brief needs (conditions, meds, allergies, labs, vitals, encounters), FHIR R4-native (feeding D9 directly). ED encounters appear only as scattered history events with no triage/acuity context — an ED persona would force fabricated data, violating the PRD's grounding principle from day one. (3) **PRD fit:** "a primary care physician with a 20-patient day" is the PRD's own first-listed example of a well-chosen narrow user. Narrow user + crisp 90-second moment gives every latency/capability decision a measurable target.
**Invalidation:** if Synthea lab *trend* depth proves too thin for a meaningful "what changed" brief (check during Stage 1 — residual, scoped down from the original blanket caveat).

## D2. Placement: SMART-on-FHIR sidecar service — **locked** (defense required — here it is)
**Options considered:**
| | SMART sidecar | Embedded PHP module | Hybrid (module UI + sidecar) |
|---|---|---|---|
| Authorization | **Inherited from OpenEMR OAuth2/SMART scopes + single-patient compartment binding — agent can never exceed the granted scope set for the bound patient; enforcement stays in the system that owns identity** (see D2 revision 2026-07-07) | Must argue correctness of direct internal data access yourself | Two enforcement surfaces to prove |
| Trust boundary | Crisp: agent = external OAuth2 client; one boundary to defend | Agent inside the EHR process; blast radius = the EHR | Two boundaries |
| Blast radius on failure | Agent dies → EHR unaffected (co-pilot, not life support) | Bug can take down/corrupt EHR runtime | Mixed |
| Standards story | SMART v2.2.0 is **certification-capable upstream code** (inherited verbatim, not independently certified by this fork — see D2 revision 2026-07-07); the industry-standard way third-party clinical apps integrate (same mechanism as Epic/Cerner app stores) | Proprietary integration | Partially standard |
| Language/tooling | Free choice (Python agent ecosystem) | Locked to PHP 8.2/Laminas — worst agent-tooling ecosystem | Both |
| Scaling (500-bed / 300-user Q) | Scale agent tier independently; EHR untouched | Scale the whole EHR to scale the agent | Mixed |
| Latency | +1 network hop + OAuth (mitigated: token cache, colocated containers, FHIR calls parallelized) | Fastest data path | Middle |
| Wk2–3 compounding | Agent service is EHR-agnostic at the FHIR layer | Married to OpenEMR internals | Middle |
**Why sidecar wins:** the only option where the answer to "where are the trust boundaries and how are they enforced?" is *"in OpenEMR's own certified authorization layer — we didn't build a parallel one to get wrong."* The embedded module's single advantage (raw data-path speed) is the one thing we can mitigate (caching, parallel calls); its disadvantages (authz burden, blast radius, PHP lock-in) are the things we can't afford.
**Regulatory backing (R5):** ONC certification § 170.315(g)(10) *requires* certified EHRs to expose FHIR R4/US Core with SMART app launch — this integration pattern is the federally mandated one, which also makes the agent portable to any certified EHR beyond OpenEMR.
**Tradeoffs owned:** OAuth2/SMART plumbing is upfront cost; latency budget spent on a hop. Both measured and mitigated, not hidden.

**Revision 2026-07-07 (audit F-A.2, F-S.1, F-S.2, F-S.5).** The audit CONFIRMED D2's substance and corrected its wording:
- *Confirmed:* OpenEMR exposes a real SMART-on-FHIR / OAuth2 EHR-launch surface with **S256-enforced** PKCE (`CustomAuthCodeGrant.php:53,253-270`, not merely advertised) and `authorization_code` tokens attribute FHIR calls to the actual clinician in `api_log.user_id` (F-A.2, F-S.5). The external-sidecar / inherited-authz boundary is sound. *Provenance note:* this is certification-**capable** stock upstream OpenEMR (imported verbatim in `ef3d490`), not fork-authored or independently certified — the "*certified* in this codebase" phrasing above should be read as "certification-capable upstream code inherited unchanged."
- *Corrected wording:* the prior "*Inherited from OpenEMR OAuth2/SMART scopes — enforcement stays in the system that owns the ACLs*" overstated the GACL role. For **patient-scoped** SMART tokens (the agent's model), enforcement is **granted scopes + single-patient compartment binding**, with GACL adding nothing for that bound patient: the FHIR service unconditionally overwrites the `patient` param with the server-derived puuid (`FhirConditionService.php:88`, `FhirServiceBase.php:227-231`), so no cross-patient read is possible, but the ACL `aclCheckCore` loop is skipped on the patient-request branch (`FhirGenericRestController.php:94-102`). The agent's ceiling is therefore the **granted scope set**, not the union of scope and ACL — request the narrowest scopes possible and do not treat ACL as a second containment layer under patient scopes.
- *Caveat (F-S.2):* OpenEMR's own `checkUserHasAccessToPatient()` is a stub returning `true` (`BearerTokenAuthorizationStrategy.php:479-485`) — the server does not independently verify the clinician is authorized for the launch patient. The **agent-side session pin (D12) is the real clinician↔patient guarantee**, not an inherited server check; do not weaken it.

## D4. LLM: Claude Sonnet 4.6 primary + Haiku 4.5 utility, thin provider abstraction — **locked** (delegated; research-backed)
**Why (vs GPT-5.5):** Sonnet 4.6 $3/$15 vs GPT-5.5 $5/$30 per M tokens — ~40–50% cheaper at equivalent capability class for structured tool use; **prompt caching (90% off cache hits) matches our exact traffic shape** (same patient-context prefix re-sent every turn); 200K-class context fits full patient summaries without truncation gymnastics. Haiku ($1/$5) handles cheap structured utility calls. Assumed BAA per PRD applies equally to both vendors.
**Why not open source (now):** no GPU anywhere in our managed-platform deployment; unproven clinical tool-use reliability inside one week; PRD explicitly permits provider+BAA. **Revisited at 100K users** in cost analysis where self-hosted inference flips the economics — this is the "architectural change at scale," not a per-token multiplication.
**Abstraction:** one internal `llm.complete()` seam so provider swap is a config change — supports the cost-analysis narrative and vendor-risk answer.
Sources in RESEARCH.md R1.

This makes keeping Langfuse in-boundary (the original D5 rationale) doubly load-bearing: it holds the only complete audit record of *which app under which grant* touched PHI. *(The "in-boundary" framing is superseded by the 2026-07-08 revision — the accountability role itself is unchanged.)*

**Unchanged:** the 2026-07-07 elevated role — Langfuse remains the §164.312(b) accountability record / system of record for `{client_id, exercised_scopes, correlation_id}` per FHIR call; that role concerns *what is logged*, not *where it runs*. ## D6. Agent loop: direct Anthropic SDK tool-use loop, no LangChain/LangGraph — proposed
**Why:** one agent, one loop, ~6 tools — a framework adds abstraction debt and obscures the trace story we're graded on. Direct loop = every step visible, every schema ours (Pydantic), trivial correlation-ID threading. Defense: "we can explain every line between the user and the model."
**Research backing (R6):** the 2026 consensus framing is "who owns the loop" — direct SDK loops are the recognized right answer for single-agent, single-provider systems (streaming/caching/tool-use with nothing hidden); LangGraph earns its complexity for multi-stage multi-provider stateful graphs, which this is not. Our case sits squarely in the documented direct-loop category — this is a positioned choice, not naïveté.
**Invalidation:** wk2–3 multi-agent requirements (migration seam: tool registry + orchestrator interface stay framework-shaped).

## D7. Verification layer: evidence-packet + structured-claims pipeline — **locked** (v2, revised 2026-07-06 after prior-art review; supersedes v1 citation-tags-in-prose design)
*Patterns adapted from studying a completed prior clinical-copilot implementation — concepts adopted, no text or artifacts reused; every mechanism below is ours to build and defend.*
1. **EvidencePacket:** every tool result is normalized into evidence records with stable IDs (`ResourceType:resource_id:hash8`). The packet — not raw FHIR — is what the LLM sees and what the verifier resolves against. Citations become concrete lookups, not string matching.
2. **Typed structured claims:** the model must answer in typed claim payloads (`MedicationClaim{name, dose, status}`, `LabValueClaim{loinc, value, unit, date}`, `AllergyClaim{substance, status}`, …), each carrying `evidence_ids` (Pydantic-enforced — doubles as the PRD's schema-contract requirement).
3. **Field-level verification:** the verifier compares claim fields against the cited evidence record's fields. **Reject on contradiction, not on absence** — missing dose in both claim and evidence is fine; claim says 10mg while evidence says 5mg is rejected.
4. **Deterministic templater:** the text the physician reads is *re-rendered from the verified structured fields*; the LLM's own prose for a claim is discarded if it diverges. The model cannot phrase its way past verification. Missing fields render as "not specified in record," never invented.
5. **Domain constraint + phrasing rules:** allergy-vs-prescription contradiction, dosage bounds, stale-lab flags (v1 rules), plus a **forbidden-phrasing screen** (an empty allergy query must never render as "NKDA"/"no known allergies" — only "no allergy records returned; confirm with patient"; no "labs are normal" without interpretation support) and a **treatment-verb blocklist** (start/stop/increase/prescribe/order/diagnose → refusal; "review"/"the chart shows" allowed).
6. **Verdicts:** `pass | flagged | blocked | refused(kind)` — logged per response, dashboarded, alertable.
Deterministic in the serving path (auditable, fast, testable as invariants); LLM-as-judge lives only in the eval suite.
**Research backing (R7):** ungrounded medical LLM hallucination rates exceed 60% and 45%+ of generated references in tested studies were fabricated — this layer is not optional theater; claim-level verification against retrieved evidence matches documented 2026 best practice and measurably cuts hallucinations 30–50%.
**Known limitations (stated honestly):** field-level match proves *provenance and consistency*, not perfect *synthesis* — a claim can cite and match a real record while still emphasizing the wrong thing; covered by golden-answer evals, not the serving path. Rule tables are demo-depth, extension path documented.

**Revision 2026-07-07 (audit) — verifier & phrasing rules made concrete.** The audit converted the abstract "domain constraint + phrasing rules" (item 5 above) into specific, evidence-backed rules the verification layer MUST encode. Each cites the finding that forces it:
1. **FHIR status fields are unreliable — never render verbatim (F-D.1, F-D.6).** The stock Immunization mapper inverts status via a case-sensitive `"completed" == "Completed"` bug, so all 67/67 completed vaccines return `status: not-done` + `statusReason: patient objection` (verified live). Encounter.status is hardcoded `finished` for all encounters (`FhirEncounterService.php:139`). The EvidencePacket normalizer treats FHIR `status`/`statusReason` as low-trust; the templater must never render "patient declined/refused [X]" or assert encounter state from these fields. **This is the concrete justification for the whole §5 layer** — a naïve agent would tell a physician the patient refused every vaccine.
2. **Allergy criticality is null dataset-wide — reject any criticality-based claim (F-D.4).** A real label-vs-numeric key mismatch (`FhirAllergyIntoleranceService.php:138-151`) nulls `criticality` for all 41 allergy rows; `type` is never set; statuses are constant. Never infer, rank, or deprioritize allergy risk from criticality; the verifier rejects any `AllergyCriticality` claim (no supporting evidence field exists).
3. **Empty allergy result → confirm-with-patient, never NKDA (F-D.5).** OpenEMR has no "No Known Allergies" record type; an empty AllergyIntolerance bundle is indistinguishable from affirmed NKDA. A dedicated empty-result rule is mandatory: render **"no allergy records returned; confirm with patient"** — never "NKDA" / "no known allergies." Absence is the hazard.
4. **Consume ALL conditions; never filter clinical-status=active (F-D.6).** The `clinical-status` search param maps to a non-existent SQL column and returns nothing (broken filter); resolved conditions are labeled `inactive`, not dropped, on a default read. Read all conditions; reject a "no history of X" claim if an inactive/resolved match exists.
5. **Flag stale labs; reject valueless observations (F-D.6).** Demo lab dates cluster at 2021-07-20 with `0000-00-00` zero-dates in `procedure_result`; a "recent labs" read must flag decade-stale dates rather than imply currency, and valueless observations need a non-null value or are rejected (not rendered as a finding).
6. **Empty medication dose → confirm-before-dosing; de-dup order+plan (F-D.2).** Seed `prescriptions.dosage='1.00'` yields no usable dose text (correctly suppressed by the `!is_numeric` guard); an empty-dose claim renders **"dose not specified — confirm before dosing,"** never an invented dose. MedicationRequest returns each drug twice (intent=order from `prescriptions` + intent=plan from `lists`) — the EvidencePacket de-duplicates to one stable ID per drug.

**Decision:** all services deploy as a Railway project: OpenEMR (Docker image + volume for `sites/`), Railway managed MySQL, agent service (FastAPI, native build). *(Revised 2026-07-08 with D5: Langfuse services removed from the project — observability is Langfuse Cloud, external under an assumed BAA. Native GitHub integration deploys on push and can wait on CI checks → our eval-gate-before-deploy story survives (GH Actions runs evals; Railway deploys only on green).
**Tradeoffs owned (say these before the interviewer does):**
- *No known OpenEMR-on-Railway prior art* — we pathfind; mitigation: OpenEMR production image + volume is the documented pattern for single-host installs, and local compose remains the fallback dev truth.
- *Platform reliability:* Railway had several major outages in 2025–26 (incl. an 8-hour event, May 2026); mitigation: for a real hospital deployment the answer is multi-region/self-hosted — documented as a scale-up architectural change in the cost analysis, not hidden.
- *Usage-based cost variance*; mitigation: monitor from day one via Railway metrics. *(Revised 2026-07-08: the original worst driver — memory-hungry self-hosted ClickHouse — is gone with the D5 flip to Langfuse Cloud; remaining variance is OpenEMR + MySQL + agent only.)*
- *Same-infra rule honored:* fork AND agent both live in the one Railway project (PRD Stage 2 requirement).
**Rollback:** Railway redeploys any previous deployment in one click; plus `git revert` → auto-redeploy.
**Rejected alternative (VPS + Compose + Caddy):** strongest raw-control/cost story (~$13–25/mo fixed) and compose parity, but consumes the week's scarcest resource on undifferentiated ops. Rejected outright — Railway is the deployment method, full stop. Operational contingency is Railway-native: one-click redeploy of any previous deployment, plus local Docker Compose for dev parity and demo continuity.

## D9. Data access: FHIR API only (no direct SQL from agent) — **locked**
All agent reads go through OpenEMR's FHIR API with the user-delegated token. Direct SQL would bypass ACL + audit logging and forfeit D2's entire defense. Cost: FHIR verbosity/latency → mitigated with per-session TTL cache + parallel calls. (Direct SQL remains available to the *audit* work, which is read-only analysis, not the serving path.)

**Addendum 2026-07-09 (scope & refresh posture — minimum-necessary, decided during E3).** The agent requests exactly the **minimum-necessary** scope set: `openid` (for the clinician OIDC `sub` that pins the session, D12) + the **six** FHIR read scopes its tools use — `user/{Patient,Condition,MedicationRequest,AllergyIntolerance,Observation,Encounter}.read`. Verified live (E2/E3 re-consent) that OpenEMR grants all six; a runtime guard (`assert_required_scopes_granted`) fails at token exchange if any is missing, so the meds/labs/encounter tools never 401 mid-turn (granted ≠ requested — we check the *granted* set).
- **`offline_access` DROPPED — deliberate, not accidental.** The session is bounded by MIN(token expiry, idle timeout, turn cap) with re-launch on expiry (§3a/D12), so a refresh token buys almost nothing while adding a long-lived credential that widens the PHI-access surface. Minimum-necessary (F-C.5, the covered entity's own duty, which a BAA does not discharge) says don't request it. **The agent does not store or use a refresh token; a mid-session token expiry prompts re-launch, not silent refresh.**
- **`api:oemr` / `api:fhir` DROPPED** — FHIR reads succeed on `user/*.read` alone (verified live), so the broad `api:*` scopes are not minimum-necessary.
- **ARCHITECTURE §6 reconcile flag:** the §6 failure-mode row currently reads "FHIR 401 mid-session → refresh_token grant, else prompt re-launch"; under this addendum the refresh branch is removed → **"re-launch on expiry."** Route the §6 wording change through /arch-finalize (not edited in-code).

## D10. Parallel tool execution — **locked**
All FHIR tool calls with no data dependency execute concurrently (`asyncio.gather` w/ per-call timeouts + a total turn budget). The pre-visit brief's 6 baseline reads (demographics, conditions, meds, labs, encounters, allergies) are independent → wall-clock ≈ slowest call, not the sum. Dependent chains (e.g., encounter → its observations) remain sequential by necessity and are documented per tool. This is the primary latency lever behind the "seconds, not minutes" requirement and gets measured per-stage in Langfuse spans (parallel fan-out visible in traces — good demo material).
**Correlation across boundaries:** every outbound FHIR call carries the request's correlation ID as an `X-Copilot-Request-Id` header. *(Original claim, superseded 2026-07-07: "so agent traces can be joined against OpenEMR's own API/audit log — the 'full trace reconstructable from logs alone' requirement holds across service boundaries.")*

**Revision 2026-07-07 (audit F-C.1, F-C.2, F-A.5, F-P.6) — REQUIRED correction.** The shared-id join into OpenEMR's `api_log` is **not achievable** and the prior claim is withdrawn. Verified: `api_log` has no correlation/request-id column and **no code path persists an inbound header** — `ApiResponseLoggerListener.php:77-86` builds the log row purely from session + request line and never reads request headers; the fixed 7-key `ApiData` DTO has no header slot; adding a column is disallowed (D9 read-only, no app modification). Restated design:
- **Langfuse (D5) is the authoritative agent-side trace and the system of record** for `client_id` + granted scopes + correlation id (D5 revision). The "full trace reconstructable from logs alone" requirement is met **on the agent side**, not via a hard cross-system join.
- **api_log correlation is best-effort / fuzzy** on the natural key `(user_id, patient_id, request_url, utc_timestamp)` — and it is *weak*, because every agent call logs the **same delegated `user_id`** (and `api_log.patient_id` was observed `0` for user-role tokens, F-S.5), so the tuple rarely disambiguates concurrent turns. The agent still sends `X-Copilot-Request-Id` (cheap, forward-compatible if OpenEMR ever persists it), and records `{correlation_id, client_id, scopes, user, patient, request_url, utc_timestamp}` per call so a compliance query can re-derive the corresponding api_log row by URL + timestamp. Do **not** promise a hard join in §7 or the dashboard.
**Tradeoffs owned:** concurrent load on OpenEMR's PHP/MariaDB (cap fan-out, verify under k6); error aggregation complexity (partial-failure path already specified, F3).
**Latency calibration (honesty note, re-tagged 2026-07-07 → R12):** an *anecdotal prior-art observation* (a completed prior implementation of this same case study) put p50 ≈ 28s end-to-end with the LLM ≈ 85% of wall-clock. **This is an unverified planning assumption, not a sourced number (R12)** — no agent exists yet to measure, and the audit measured only OpenEMR's FHIR floor (≈0.39s/read live, F-P.5). Our "seconds" story is defended at the *perceived*-latency level (streaming-first UX, first tokens ~2–3s; per-stage budgets; prompt caching) and the *shape* (LLM dominates wall-clock, so streaming is the lever), with the 28s figure **replaced by measured Langfuse data at Early submission** — the re-baseline is the primary story, the prior-art number is directional only.

## D12. Clinical safety posture: explicit non-goals, hard-stops, refusal-as-feature — **locked** (added 2026-07-06)
**Non-goals (stated, not implied):** the agent does not diagnose, recommend treatment, prescribe, order labs/imaging, message patients, write/edit chart notes, search across patients, or use write scopes. Read-only by construction, not by policy.
**Deceased-indicator hard-stop:** before any summarization, a pre-flight check on deceased indicators; if set → deterministic refusal ("this record contains a deceased indicator — review the chart manually"). Applies to all use cases; refusing across the board beats per-use-case cleverness.
**Refusal-as-feature:** canonical, deterministic refusal messages for: ambiguous data resolution ("I could not identify a prior visit reliably — review the chart manually"), unauthorized/wrong-patient requests, treatment-advice requests, and expired sessions (re-launch from chart). Rationale: in a clinical setting, a confident wrong answer is the trust-killing failure mode (PRD's own "why this matters"); a clear refusal preserves trust and is *the* defensible behavior.
**Session posture:** session pinned to (clinician, patient) at creation — patient switch requires a fresh launch; turn cap per session bounds cost runaway and PHI exposure window.

**Revision 2026-07-07 (audit F-S.7, F-D.5) — eval-fixture requirement added (closes a test gap).** Two safety-critical paths **cannot be exercised against the demo data** and would otherwise ship unverified: the deceased-indicator hard-stop (all 25 Synthea patients are alive — `SELECT COUNT(*) FROM patient_data WHERE deceased_date IS NOT NULL AND deceased_date != '0000-00-00…'` = 0), and the empty-allergy → confirm-with-patient path (the canonical patients all have allergy records). **Requirement:** the eval suite (§8) MUST inject synthetic fixtures for both — a deceased-patient fixture (audit-only DB `deceased_date` on one patient, or a mocked FHIR `Patient.deceasedDateTime` payload) driving the deterministic-refusal branch, and a no-allergy-record fixture driving the forbidden-phrasing rule (D7 rev. rule 3). The deceased hard-stop keys on `Patient.deceasedDateTime` OR `deceasedBoolean == true` (the exact FHIR fields, `FhirPatientService.php:741-751`). Without these fixtures, D12's two most important guarantees are asserted but untested — a happy-path-only gap the PRD explicitly fails.

**Addendum 2026-07-09 (verify-in-loop review) — label-fallback on absence, a documented known limitation.** When a claim cites a *real, resolved* record whose **label** field is empty (a `MedicationRequest` with no `name`, an `Observation` with no `display`), the field-level check finds no *contradiction* (absence, not mismatch) → the claim passes on that field, and the deterministic templater renders the claim's own label. This is the general "provenance-and-consistency, not synthesis" limitation made concrete: it lets the model attach a label to a labelless-but-real record. Bounded and non-critical because the **sensitive fields — medication `dose`, lab `value` — are always record-sourced, never the claim's** (F-D.2 holds; no dose/number is ever invented), and real records carry labels (edge case). **Decision:** accept as a documented §5 known-limitation (recorded in ARCHITECTURE §5), **pin the current behavior with a regression test** so any change is conscious, and **defer** the hardening (render/annotate only a label the cited record actually carries) to E6-verifier v2. Surfaced by the T-E6a independent reviewer; not attributable to the verify-in-loop wiring.

## D13. Deterministic degradation on LLM failure — **locked** (upgrades the §6 "proposed" fallback)
When the LLM hard-fails (retries exhausted / hard timeout), the agent renders the EvidencePacket directly through the same deterministic templater (D7.4): grouped by category, values + dates, state-aware phrasings, an explicit "generated without LLM assistance — records present, clinical synthesis is not" banner. The verifier still runs. The physician always gets something grounded — never "LLM failed, no answer." Fallback rate is traced and alertable.

## D14. OAuth app provisioning: user-scoped apps register DISABLED — **locked** (added 2026-07-07, audit F-S.6)
The agent is a confidential SMART client requesting `user/*` scopes (D2/D9). The audit verified that OpenEMR registers such clients **disabled** by default: `ClientRepository.php:78-88` + `ScopeRepository.php:334+` set `is_client_enabled=0` when a confidential client requests `user/*` or `system/*` scopes (only `patient/*`-only clients auto-enable), and `oauth_app_manual_approval=0` does not change this. Registration therefore ≠ usable — the agent cannot mint tokens until an OpenEMR admin enables it in **Administration → API Clients**.
**Requirement:** the deployment runbook MUST include the one-time "register the SMART app, then enable it in Administration" step, before any agent token flow is exercised (a silent 401/"client disabled" is otherwise the first-run failure). Mild defense-in-depth positive: an attacker cannot dynamically register a user-scoped app and use it without admin action. Posture flag: if the design ever pivots to `patient/*` (patient-facing) scopes, apps auto-enable with no admin gate — revisit then.

## D15. api_log is a second in-boundary PHI store — **locked** (added 2026-07-07, audit F-S.4, F-C.3)
The PHI inventory is not just Langfuse Cloud (D5, external under assumed BAA) + the LLM channel (D4). OpenEMR's `api_log` **ships defaulting to Full Logging** (`api_log_option=2`, `library/globals.inc.php:2900`) and writes the **full FHIR response body — complete clinical bundles — unencrypted** into `api_log.response` (`ApiResponseLoggerListener.php:62-64,83-85`; verified a 32 KB Condition bundle at rest; the encryption path was removed upstream). The agent's D10 six-read fan-out therefore duplicates each patient's allergies/meds/problems/labs into `api_log` in plaintext on **every turn**.
**Requirement:** the deployment must make an explicit **`api_log_option` / retention decision** (e.g. set `api_log_option=1` to log metadata without bodies for the agent's service account, or accept Full Logging and document api_log as an in-boundary PHI store inheriting the deployment's encryption-at-rest + retention obligations). There is **no retention/purge config** in OpenEMR and ATNA external forwarding is off (`enable_atna_audit=0`), so Full Logging accumulates PHI unbounded. The compliance section of ARCHITECTURE.md must enumerate api_log alongside Langfuse Cloud and the LLM provider as PHI stores with named incident-response owners. No agent code change (D9 read-only) — this is a deployment-config + documentation obligation.
