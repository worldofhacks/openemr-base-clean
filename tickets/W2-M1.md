## Context

The W1 agent is a FastAPI app (`agent/app/`, image `python:3.12-slim`, `pip install .`,
uvicorn on `$PORT`) that is **LIVE** on Railway: project `openemr`
(id `1bddbc72-6307-4ec9-b6dd-8184310fbdcf`), environment `production`, service `agent`,
Online at `https://agent-production-9f62.up.railway.app`. The Railway CLI is logged in
(account itsnotthatsecure@gmail.com). A broken deploy **must be rolled back** (Railway
one-click redeploy of the prior deployment) and `/health` must be green post-deploy —
never leave the live W1 agent down. Secret VALUES are never read into reports, printed,
logged, or committed; all config is env-only.

**Plan-trace substitution note** (mirrors the W2-M24 W2-OA2 substitution pattern): the
plan's "CI build stage passes with the new image" cannot literally run this epic — the
never-push rule means no GitHub Actions run occurs. AC-3's local `docker build agent/`
plus the Railway builder verdict is the substitute (also gates.md Tier-2 "Container
build"), and this substitution must be NOTED in the ticket report.

W2 adds native dependencies the W1 image lacks: the Tesseract binary + English (`eng`)
traineddata (W2-D3 — local OCR grounds every extraction), pdfium via `pypdfium2` plus
`pdfplumber` and `pytesseract` (W2-R6 reader stack — W2-M4 depends on this ticket merging
so `pdfplumber`'s pillow dependency is installed; W2-M4 may NOT touch pyproject), and the
ONNX/FastEmbed stack that loads `bge-small-en-v1.5` and the local reranker
`mxbai-rerank-base-v1` (W2-D4 rev fallback). This ticket **solely owns**
`agent/pyproject.toml` / `agent/Dockerfile` / `agent/railway.json` edits this wave.
`langgraph>=1.2,<2` is already pre-staged in pyproject (orchestrator commit `5c4e5ef`) —
do not re-add or alter it. **PyMuPDF is banned (AGPL, W2-R6 — locked-decision). No torch
anywhere in the dependency tree** — the ONNX path exists precisely to avoid it.

The Railway plan memory limit is **UNKNOWN until this ticket measures it**
(research-required — never invent the number). The ceiling rule is locked
(locked-decision): `W2_WAVE0_RSS_CEILING_MB` = at most **80% of the measured actual plan
memory limit**; peak RSS must stay under it. On failure, the documented ladder applies in
order: **quantize ONNX models → raise Railway service memory → externalize the index
(last resort)** — and the local-reranker stack is blocked for feature work until the
ladder resolves it (§6 W2-O1).

`agent/ops/spike_rss.py` is a NEW operator CLI (ops scripts may print — see
`.tdd-swarm/gates.md`) that runs **inside the deployed Railway container**: it reads the
container memory limit (cgroup v2 `memory.max` or equivalent), then concurrently holds
(a) bge-small embeddings, (b) `mxbai-rerank-base-v1`, (c) a representative built hybrid
index (synthetic, non-PHI, on the order of several hundred chunks per §2 corpus sizing),
and (d) one 200-DPI Tesseract OCR page (synthetic generated image), while exercising one
HTTP request against the app, sampling cold and peak RSS, and printing a report.
Proposed: the probe may add `rank-bm25` to pyproject if it builds the BM25 side of the
index; exact distro package names for Tesseract (Debian slim: `tesseract-ocr`,
`tesseract-ocr-eng`) and whether ONNX models are baked at image build vs fetched at probe
time are the implementer's call — record the choice and its cold-start/image-size impact.

## Acceptance Criteria

- **AC-1**: Given the W2 dependency set installed per `agent/pyproject.toml`, when the
  extended import smoke tests in `agent/tests/test_app_boot.py` run, then `pypdfium2`
  imports and reports a version, `pdfplumber` imports (the dep this ticket stages for
  W2-M4 — plan-review minor finding), `pytesseract` imports and resolves a working
  `tesseract` binary on PATH (version call succeeds), and the FastEmbed/ONNX modules
  import — with **no model download and no network access** in the tests.
- **AC-2**: Given the installed environment, when the dependency tree is inspected
  (`importlib.util.find_spec("torch")` plus an installed-distributions scan via
  `importlib.metadata`), then **no torch distribution is present anywhere in the
  dependency tree** and `pip check` on the environment is clean. (Guard test: green at
  RED phase by construction; it pins the invariant as deps land.)
- **AC-3** [live-measure]: Given the extended Dockerfile, when the image is built locally
  and deployed to Railway service `agent`, then both builds succeed and `/health` is
  green post-deploy with the new deps loaded; if any dep fails on Railway's builder, the
  deploy is rolled back (W1 agent stays live, `/health` green on the rolled-back
  deployment) and the fallback investigation is documented before feature work proceeds
  — if no failure occurs, "builder green" is recorded.
- **AC-4** [live-measure]: Given the deployed container, when checked in-container, then
  `tesseract --version` runs, `eng` traineddata is resolvable (listed by
  `tesseract --list-langs` or equivalent), `pypdfium2` renders a page, and bge-small ONNX
  loads under FastEmbed — evidence recorded in the ticket report.
- **AC-5** [live-measure]: Given the deployed Railway container, when
  `agent/ops/spike_rss.py` concurrently holds bge-small + `mxbai-rerank-base-v1` + the
  representative built hybrid index + one 200-DPI Tesseract OCR page while exercising one
  request, then the **actual plan memory limit**, cold RSS, and peak RSS are recorded;
  `W2_WAVE0_RSS_CEILING_MB` is set to at most 80% of the measured plan limit; and peak
  RSS stays under the ceiling. Failure blocks the local-reranker stack and invokes the
  quantize → raise-memory → externalize-index ladder before feature work (named failure
  path, locked-decision).
- **AC-6** [live-measure]: Given the new image, when compared against the prior W1 image,
  then the image-size delta and cold-start time are recorded in the ticket report
  (W2-R4 planning numbers become measured baselines).
